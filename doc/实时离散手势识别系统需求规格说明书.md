# 实时离散手势识别系统 — 需求规格说明书

> 项目代号：RT-Gesture（Real-Time Discrete Gesture Recognition）
> 版本：v0.3 DRAFT

## 1. 项目概述

### 1.1 目标

基于 generic-neuromotor-interface 项目的 Discrete Gestures 分支，构建一个**实时离散手势识别系统**。系统从 EMG 数据流（当前阶段为离线数据模拟）中实时推断用户手势，并通过桌面 GUI 展示实时监控结果。

### 1.2 工作路径

```
imporve-generic-neuromotor-interface/workspace/
```

### 1.3 项目定位

- **当前阶段**：离线数据模拟实时流 → 实时推理 → 结果展示
- **未来扩展**：可对接真实硬件设备的实时 EMG 数据流

## 2. 功能需求

### 2.1 核心功能

#### F1 — 离线数据实时模拟器

| 属性 | 规格 |
|------|------|
| 数据源格式 | 原项目 HDF5 文件（discrete_gestures 数据集） |
| 模拟方式 | 按原始采样率 (2000Hz) 逐块推送数据，还原真实时间节奏 |
| 通道数 | 16 通道 sEMG |
| 采样率 | 2000 Hz |
| 输出 | 通过 ZeroMQ 发布 EMG 数据块到推理进程 |
| 附加信息 | 同步推送 ground-truth prompts（手势标签时间戳），供验证对比 |

#### F2 — 实时流式推理引擎

| 属性 | 规格 |
|------|------|
| 模型架构 | DiscreteGesturesArchitecture（复用原项目） |
| 架构细节 | ReinhardCompression → Conv1d(16→512, k=21, s=10) → ReLU → Dropout → LayerNorm → LSTM(512×3层) → LayerNorm → Linear(512→9) |
| 参数量 | ~6.5M |
| 推理方式 | **滑动窗口 + LSTM 隐藏状态维持**：每收到一批新数据后，仅对新数据做前向传播，LSTM 隐藏状态跨批次保持连续 |
| 输出频率 | 200 Hz（每 5ms 产出一帧 9 类 logits） |
| 设备策略 | GPU 优先（CUDA），不可用时自动回退 CPU |
| 模型权重 | 离线训练产出的 checkpoint 文件，运行时冻结不更新 |

#### F3 — 事件检测与后处理

| 属性 | 规格 |
|------|------|
| 阈值检测 | Sigmoid 概率上穿 threshold=0.35 视为事件触发 |
| 置信度拒识 | 事件峰值概率 < rejection_threshold（默认 0.6）时丢弃该事件，避免低置信度误触发 |
| 去抖规则 | 同类事件最小间隔 debounce=50ms |
| 休息态抑制 | 所有通道概率均 < rest_threshold（默认 0.15）时标记为静息态，不产生任何事件 |
| 手势类别 | 9 类（见 §3.1） |
| 输出事件 | `(gesture_name, timestamp, confidence)` 元组 |

> **设计说明**：三层过滤机制——休息态抑制（全局静默）→ 阈值上穿检测（候选事件）→ 置信度拒识（过滤弱事件）→ 去抖（消除重复），逐步减少误激活。阈值参数均可通过 YAML 配置调整。

#### F4 — 事件输出

| 输出方式 | 说明 |
|----------|------|
| 控制台打印 | 实时打印检测到的手势事件（名称 + 时间戳 + 置信度） |
| 事件回调 | 提供 Python 回调接口，下游系统可注册监听 |
| ZeroMQ 发布 | 推理进程通过 ZMQ PUB 广播事件，GUI 或其他消费者 SUB 订阅 |

#### F5 — 桌面 GUI 实时监控

| 属性 | 规格 |
|------|------|
| 技术栈 | PyQt6 / PySide6 |
| 显示内容 | ① EMG 16 通道实时波形 ② 9 类手势概率条形图/热力图 ③ 检测事件日志列表 ④ 系统状态（FPS、延迟、设备信息） |
| 配置面板 | 启动前选择：数据源文件、模型 checkpoint 路径、设备（GPU/CPU）、阈值参数 |
| 操作控件 | 启动/停止推理、暂停/恢复显示 |

#### F6 — 离线训练管线

| 属性 | 规格 |
|------|------|
| 训练数据 | 原项目 discrete_gestures 离线数据集（HDF5） |
| 模型架构 | DiscreteGesturesArchitecture（与推理一致） |
| 训练框架 | PyTorch (+ 可选 Lightning) |
| 数据增强 | RotationAugmentation 等（复用原项目策略） |
| 损失函数 | BCEWithLogitsLoss + FingerStateMask |
| 评估指标 | CLER（与原项目对齐），MulticlassAccuracy |
| 产出 | checkpoint 文件供 F2 加载 |
| 基准对齐 | 训练/评估结果应与原项目测试基准可对比 |

#### F7 — 日志与结果存储

| 属性 | 规格 |
|------|------|
| 事件日志 | 每次运行生成带时间戳的事件日志文件（JSON Lines 格式） |
| 预测结果 | 逐帧概率 + 检测事件完整存储，含时间戳，便于离线回溯分析 |

## 3. 非功能需求

### 3.1 延迟

| 指标 | 要求 |
|------|------|
| 端到端延迟 | 从手势发生到系统输出预测 ≤ **200ms** |
| 推理延迟 | 单帧模型前向传播 << 5ms（200Hz 输出频率所需） |
| GUI 刷新率 | ≥ 30 FPS 波形更新 |

**延迟预算分解：**

| 环节 | 预算 | 说明 |
|------|------|------|
| 数据采集/模拟推送 | ≤ 5ms | 按 stride=10 样本（5ms）为一个推送周期 |
| ZMQ 传输（数据→推理） | ≤ 2ms | 本地 TCP 传输 |
| 模型前向推理 | ≤ 5ms | Conv1d + LSTM + Linear 单帧 |
| 事件检测 + 后处理 | ≤ 3ms | 阈值检测 + 拒识 + 去抖 |
| ZMQ 传输（推理→GUI） | ≤ 2ms | 本地 TCP 传输 |
| GUI 渲染 | ≤ 33ms | 30 FPS 帧间隔 |
| **模型固有算法延迟** | **~50-100ms** | Conv left_context (10ms) + 事件检测响应时间 + 去抖窗口 |
| **端到端总计** | **≤ 150ms** | 含算法固有延迟，在 200ms 预算内留有余量 |

> Phase 1 中须内置延迟打点机制（在 ZMQ 消息中携带时间戳），用于实测验证各环节是否在预算内。

### 3.2 跨平台兼容

| 平台 | 要求 |
|------|------|
| Linux | 主开发平台，完整支持 |
| Windows | 需支持 Windows + CUDA 环境运行 |

### 3.3 稳定性

- 预期连续运行时长：**分钟级**（演示/实验场景）
- 无内存泄漏（分钟级运行内存增长可忽略）
- 进程异常退出时资源正确释放（ZMQ socket、GPU 显存等）

### 3.4 数据中断与异常处理

| 场景 | 检测方式 | 处理策略 |
|------|----------|----------|
| EMG 数据流中断 | 推理进程超时未收到数据（timeout 可配，默认 500ms） | 标记中断状态，暂停事件输出，GUI 显示警告 |
| 数据流恢复 | 收到新数据包 | **重置 LSTM 隐藏状态为零**，丢弃过渡期前 N 帧输出（N = warm_up_frames，默认 20 帧 = 100ms），避免污染状态产生误预测 |
| ZMQ 消息丢包 | 通过消息序列号检测间隙 | 记录丢包日志；若连续丢包超过阈值，触发 LSTM 状态重置 |
| 推理进程崩溃 | GUI 进程检测 heartbeat 缺失 | GUI 显示错误状态，支持手动重启推理进程 |

> **关键约束**：LSTM 是有状态模型，数据不连续时隐藏状态被污染，后续所有预测都不可信。因此**任何数据中断都必须触发状态重置 + warm-up**。

### 3.5 多进程优雅关闭

系统包含 3 个进程，需定义确定性的关闭顺序，避免端口占用、显存泄漏和僵尸进程。

**关闭流程：**

1. 用户触发关闭（GUI 关闭按钮 / Ctrl+C / SIGINT）
2. GUI 进程发送 `SHUTDOWN` 控制消息到推理进程
3. 推理进程收到后：释放模型和 GPU 显存 → 发送 `SHUTDOWN` 到数据模拟器 → 关闭 ZMQ sockets → 退出
4. 数据模拟器收到后：停止数据推送 → 关闭 HDF5 文件句柄 → 关闭 ZMQ sockets → 退出
5. GUI 进程确认其他进程退出后关闭自身

**异常保护：**
- 各进程注册 `atexit` + `signal` handler（SIGINT/SIGTERM）确保资源释放
- 超时强制终止（默认 5s 等待后 force kill）
- ZMQ context 使用 `ctx.term()` 确保 socket 清理

### 3.6 测试

| 测试类型 | 覆盖范围 |
|----------|----------|
| 单元测试 | 数据模拟器、推理引擎、事件检测（含拒识逻辑）、ZMQ 消息序列化各模块 |
| 集成测试 | 端到端管道（模拟数据 → 推理 → 事件输出） |

| 关闭测试 | 验证三进程优雅关闭顺序，无端口残留/显存泄漏 |

## 4. 系统架构设计

### 4.1 多进程管道架构

```
┌─────────────┐    ZMQ PUB     ┌─────────────┐    ZMQ PUB     ┌─────────────┐
│  数据模拟器  │ ─────────────→ │  推理引擎    │ ─────────────→ │  GUI 展示    │
│  (Process 1) │   EMG chunks   │  (Process 2) │   events +     │  (Process 3) │
│              │                │              │   probabilities│              │
│ HDF5 → 2kHz │                │ Model infer  │                │ PyQt6 实时   │
│ 定时推送     │                │ + 事件检测    │                │ 波形+事件    │
└─────────────┘                └─────────────┘                └─────────────┘
       │                              │                              │
       └──────── ZMQ 也推 EMG ────────┘──── GUI 也订阅 EMG 用于波形 ─┘
```

**进程划分：**

| 进程 | 职责 | ZMQ 角色 |
|------|------|----------|
| **DataSimulator** | 读取 HDF5，按 2kHz 节奏推送 EMG 数据块 + ground-truth | PUB (emg_topic, gt_topic) |
| **InferenceEngine** | 订阅 EMG 数据，流式推理，事件检测，发布结果 | SUB + PUB (event_topic, prob_topic) |
| **GUIMonitor** | 订阅 EMG + 概率 + 事件，实时展示 | SUB |

**ZMQ 端口规划：**

| 通道 | 地址 | 内容 |
|------|------|------|
| EMG 数据流 | `tcp://localhost:5555` | 16ch EMG 数据块 (numpy array) |
| Ground-truth | `tcp://localhost:5556` | 手势标签时间戳 |
| 推理结果 | `tcp://localhost:5557` | 概率帧 + 检测事件 |
| 控制通道 | `tcp://localhost:5558` | 生命周期控制消息（SHUTDOWN / HEARTBEAT） |

### 4.2 ZMQ 消息格式规范

所有 ZMQ 消息采用统一的 **Header + Payload** 结构，使用 **msgpack** 序列化（不使用 pickle，避免安全隐患）。

**消息 Header（所有消息类型共享）：**

```python
header = {
    "version": 1,           # 协议版本号，用于向前兼容
    "seq": uint64,          # 单调递增序列号，用于丢包检测
    "timestamp": float64,   # 发送端 time.monotonic() 时间戳，用于延迟测量
    "msg_type": str,        # 消息类型标识
}
```

**各通道消息定义：**

| 通道 | msg_type | Payload 内容 |
|------|----------|-------------|
| EMG 数据流 | `"emg_chunk"` | `{"data": np.ndarray (16, N), "sample_offset": int}` — N 为本次推送的采样点数，sample_offset 为在完整数据流中的起始位置 |
| Ground-truth | `"ground_truth"` | `{"prompts": [{"gesture": str, "time": float}, ...]}` |
| 概率帧 | `"probabilities"` | `{"probs": np.ndarray (9, M), "time_range": (float, float)}` — M 为本批输出帧数 |
| 检测事件 | `"gesture_event"` | `{"gesture": str, "time": float, "confidence": float}` |
| 控制 | `"shutdown"` / `"heartbeat"` | `{}` |

**numpy 数组传输**：使用 ZMQ multipart message，Header 用 msgpack 打包为第 1 帧，numpy 数组用 `array.tobytes()` 作为第 2 帧（附带 dtype + shape 元信息在 Header 中），避免 numpy 序列化开销。

**丢包检测**：接收端检查 `seq` 连续性，若发现间隙则记录日志。推理进程连续丢失 ≥ 3 个 EMG 包时触发 LSTM 状态重置（见 §3.4）。

### 4.3 流式推理详细设计

**滑动窗口 + LSTM 状态维持策略：**

1. Conv1d 层需要 `left_context = 20` 个历史采样点（kernel_width - 1）
2. 每次收到新的 EMG 数据块（如 N 个采样点）：
   - 将 left_context 个历史点与新数据拼接：`[history[-20:], new_data]`
   - 通过 Conv1d → ReLU → Dropout → LayerNorm 得到新帧特征
   - 将新帧特征送入 LSTM，传入上一次的 `(h, c)` 隐藏状态
   - LSTM 输出通过 LayerNorm → Linear 得到 9 类 logits
   - 保存更新后的 LSTM `(h, c)` 状态供下次使用
3. 对 logits 做 Sigmoid → 阈值检测 → 去抖 → 输出事件

**推送粒度建议：** 每 10 个采样点（5ms, 即 stride=10）推送一次，与模型输出帧率对齐。或按 20ms (40 样本) 的批次推送以降低 IPC 开销。

## 7. 分阶段交付计划

### Phase 1 — 核心推理管道（MVP）

**目标**：离线数据 → 实时推理 → 控制台事件输出

- [ ] 项目骨架搭建（目录结构、配置、常量）
- [ ] 复用模型架构 `DiscreteGesturesArchitecture`，适配流式推理（LSTM 状态维持）
- [ ] 实现 `InferenceEngine`（加载 checkpoint、逐帧推理）
- [ ] 实现 `EventDetector`（阈值 + 置信度拒识 + 去抖后处理）
- [ ] 实现 `DataSimulator`（HDF5 读取 + 2kHz 定时推送）

## 8. 关键设计决策记录

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 推理策略 | 滑动窗口 + LSTM 状态维持 | 保持时序连续性，避免重复计算 |
| 进程间通信 | ZeroMQ | 高效、跨平台、松耦合，适合多进程管道 |
| GUI 框架 | PyQt6/PySide6 | 跨平台原生桌面 UI，与 pyqtgraph 集成好 |
| 配置管理 | 纯 YAML（不用 Hydra） | 轻量级，GUI 易于读写，减少依赖 |
| 模型权重 | 离线训练后冻结，运行时不更新 | 稳定可预测，简化实时系统复杂度 |
| 数据输入 | HDF5 模拟实时流（当前阶段） | 利用现有数据快速验证，后续可替换为真实设备 |
| 部署平台 | Linux + Windows (CUDA) | 用户明确要求 Windows 支持 |
| 交付方式 | 分阶段（核心→GUI→训练） | 降低风险，尽快拿到可运行的 MVP |
| 消息序列化 | msgpack（不用 pickle） | 跨语言兼容、无任意代码执行风险、性能优于 JSON |
| 事件后处理 | 三层过滤（休息态+拒识+去抖） | 显著减少误激活，提高实用性 |
| 数据中断处理 | LSTM 状态重置 + warm-up | 避免污染状态产生误预测 |

## 10. 风险与待确认事项

| 风险/待确认 | 说明 | 缓解策略 |
|-------------|------|----------|
| LSTM 流式推理一致性 | 滑动窗口逐帧推理与原项目整段推理的数值结果需验证一致 | Phase 1 中设计对比测试用例 |
| Windows PyQt + CUDA 兼容 | 需确认 PyQt6 + CUDA PyTorch 在 Windows 上无冲突 | 早期搭建 Windows 测试环境 |
| ZMQ 传输延迟 | 高频小包传输可能引入额外延迟 | 实测后调整推送粒度（如攒 20-50ms 数据批量发送） |
| 200ms 延迟预算分配 | 需合理分配给：数据传输 + 推理 + 事件检测 + GUI 渲染 | 建立延迟 profiling 机制 |
| 现有 checkpoint 兼容性 | 原项目 checkpoint 含优化器状态，需验证能否仅加载模型权重 | 使用 `strict=False` 或提取 state_dict |

# C04 - 稳定性与性能达标

## 1. 周期目标

使 MVP 达到需求文档的非功能指标：延迟、恢复、优雅关闭、分钟级稳定运行。

## 2. 输入与边界

输入：
- 需求文档 §3.1、§3.3、§3.4、§3.5、§3.6
- 技术文档 §16、§17、§18

本周期包含：
- 异常场景处理完善
- 延迟打点与预算验证
- 完整测试矩阵落地
- 跨平台生命周期一致性处理

## 3. 任务清单

1. 异常恢复
- 数据超时（默认 500ms）触发中断状态。
- 数据恢复后执行 LSTM 重置并丢弃 warm-up 前 20 帧。
- 连续丢包阈值触发状态重置。

2. 优雅关闭
- GUI/Main 发送 `SHUTDOWN`，推理进程转发到模拟器。
- 所有 socket `close(linger=0)`，context `term()`。
- 超时后强制终止策略（5s）。

3. 延迟测量
- 在消息 header 统一写 `timestamp(time.monotonic())`。
- 输出分环节延迟：传输、推理、后处理、总管道延迟。
- 形成基准报告并记录告警阈值。

4. 测试实现
- 单元：网络、事件检测、ZMQ、配置。
- 集成：端到端、数据中断恢复、优雅关闭。
- 基准：推理耗时、消息传输耗时、总延迟。

5. 跨平台细节
- 明确 `spawn` 启动方式。
- 处理 Windows `signal` 差异。

## 4. 交付物

1. 稳定性测试报告。
2. 延迟预算验证报告（含实测值与预算对比）。
3. 关闭流程验证记录（无端口残留/无僵尸进程）。

## 5. 验收标准

1. 端到端延迟满足 `<= 200ms`。
2. GUI 渲染前链路延迟满足需求预算（技术文档建议 `<50ms`，不含算法固有延迟）。
3. 异常恢复链路验证通过：中断 -> reset -> warm-up -> 恢复正常事件。
4. 分钟级连续运行无明显内存泄漏。

## 6. 风险与缓解

风险：
- 延迟波动大导致偶发超预算。

缓解：
- 提供 chunk_size 可调、批量推送策略与告警阈值，按实测调整。

# C01 - 环境与工程骨架

## 1. 周期目标

建立 RT-Gesture 的最小可开发基线，为后续算法与联调提供稳定工程框架。

## 2. 输入与边界

输入：
- 需求文档 §5、§6、§7
- 技术文档 §1、§2、§10、§11

本周期包含：
- 开发环境对齐（指定 conda 环境，`pip install` 规则）
- 项目目录骨架创建
- 常量与配置管理基础模块
- 日志框架初始化

本周期不包含：
- 流式推理逻辑实现
- ZMQ 实时通信联调
- GUI 与训练流程

## 3. 任务清单

1. 环境基线确认
- 固定 Python/Torch 环境与路径约束。
- 补充 `scripts/install_deps.py`（PyQt6、pyqtgraph、msgpack）。
- 明确 `pyzmq` 版本策略（建议上界从 `<27` 调整为 `<28` 并实测）。

2. 工程结构搭建
- 创建 `workspace/rt_gesture/`、`config/`、`scripts/`、`tests/` 标准结构。
- 放置 `__init__.py`、`pytest.ini`、`README` 基础骨架。

3. 基础模块落地
- `constants.py`：手势枚举、采样率、端口、消息类型、默认阈值。
- `config.py`：dataclass + YAML 映射、默认值、保存能力。
- `config/default.yaml`：集中配置，路径可覆盖。
- `logger.py`：运行日志目录和基础 logger 初始化。

4. 基础可运行性验证
- 配置加载测试。
- 常量一致性检查（9 类手势映射与原项目一致）。
- 日志目录自动创建验证。

## 4. 交付物

1. 基础目录结构完成并可导入主包。
2. `config/default.yaml` 可被加载为完整 `AppConfig`。
3. 日志系统可生成 `logs/<timestamp>/runtime.log`。
4. C01 对应单测通过（配置、常量、日志初始化）。

## 5. 验收标准

1. `pytest -k "config or constants or logger"` 全通过。
2. 任意配置缺省时能使用默认值启动基础模块。
3. 不允许出现硬编码平台路径（需支持相对路径解析）。

## 6. 风险与缓解

风险：
- 环境版本偏差导致后续 GUI/通信问题前置埋雷。

缓解：
- C01 结束即产出《环境校验记录》（Python/Torch/pyzmq/PyQt6 版本）。
